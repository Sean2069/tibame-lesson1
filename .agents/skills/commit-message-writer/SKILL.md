# Commit 訊息產生器

## 你的任務

分析目前 git repo 的變更，生成 **2–3 個候選的繁體中文 commit 訊息**，讓使用者選擇或修改後使用。

---

## 第一步：收集變更資訊

依序執行以下指令，收集足夠的上下文：

```bash
git diff --staged          # 查看已暫存的變更（最重要）
git diff                   # 查看未暫存的變更（做為補充參考）
git status                 # 了解整體狀態
git log --oneline -5       # 了解專案風格與最近做了什麼
```

**判斷邏輯：**
- 若有 staged 變更 → 以 staged 為主要分析對象
- 若 staged 為空 → 告知使用者，詢問要分析 unstaged 變更還是先暫存（`git add`）
- 若兩者都空 → 回報沒有任何變更，無法生成訊息

---

## 第二步：分析變更類型

閱讀 diff 內容後，判斷主要的變更性質。一次 commit 可能涉及多種類型，但應找出**最主要的一種**作為訊息主旨，其餘列在說明中。

### 表情符號對照表

| 表情 | 使用時機 |
|------|----------|
| ✨ | 新增功能、新特性、新頁面 |
| 🐛 | 修正 Bug、修正錯誤行為 |
| 📝 | 文件更新（README、註解、說明文字） |
| ♻️ | 重構程式碼（功能不變，改善結構） |
| 🎨 | 改善程式碼格式、排版、命名 |
| ⚡ | 效能優化 |
| 🔧 | 設定檔、環境變數、工具配置 |
| 🗑️ | 刪除無用程式碼、移除功能 |
| 🔒 | 安全性修復或強化 |
| 🚀 | 部署相關、CI/CD 設定 |
| 🧪 | 新增或修改測試 |
| 📦 | 更新相依套件（package.json、requirements.txt 等） |
| 💄 | UI 視覺樣式調整（CSS、排版、顏色） |
| 🌐 | 國際化、多語系相關 |
| 🔀 | 合併分支 |

---

## 第三步：生成候選訊息

根據分析結果，生成 **2–3 個候選訊息**，涵蓋不同的描述角度或詳細程度。

### 訊息格式

**簡潔型**（單行，適合小改動）：
```
✨ 新增使用者登入功能
```

**詳細型**（含補充說明，適合較大改動）：
```
✨ 新增使用者登入功能

- 實作 JWT 驗證機制
- 新增登入/登出 API 端點
- 加入 token 自動刷新邏輯
```

**寫作原則：**
- 主旨用**動詞開頭**（新增、修正、更新、移除、重構、優化...）
- 描述**做了什麼**，而非如何做到
- 訊息簡潔有力，主旨控制在 **50 字以內**
- 若變更涉及多個功能/模組，優先描述**影響最大的**，細節放補充說明
- 避免使用「修改了一些東西」「小改動」等模糊措詞

---

## 第四步：呈現結果

以清楚的格式呈現候選訊息，例如：

```
根據你的暫存變更，這裡有 3 個 commit 訊息候選：

**選項 1**（簡潔）
✨ 新增使用者 Email 登入功能

**選項 2**（詳細）
✨ 新增使用者 Email 登入功能

- 新增 /api/auth/login 端點
- 實作密碼雜湊與 JWT 簽發
- 前端加入登入表單元件

**選項 3**（聚焦後端）
🔧 實作 Email 認證 API 與 JWT 機制

你可以直接告訴我選哪個，或說「幫我用選項 2 commit」，我就會執行 git commit。
```

**若使用者確認後，詢問是否要直接執行：**
```bash
git commit -m "✨ 新增使用者 Email 登入功能"
```
或含多行說明的版本：
```bash
git commit -m "$(cat <<'EOF'
✨ 新增使用者 Email 登入功能

- 新增 /api/auth/login 端點
- 實作密碼雜湊與 JWT 簽發
- 前端加入登入表單元件
EOF
)"
```

**若使用者不滿意**，根據他們的回饋重新生成，不要重複完全相同的建議。
